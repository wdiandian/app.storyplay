# StoryPlay平台 Roadmap

## 使用规则

这个文档是后续开发的主控 roadmap。后续每次开发默认按下面流程执行：

1. 先更新或确认本文件里的当前阶段、目标和验收标准。
2. 再按当前阶段拆任务并开发。
3. 开发完成后更新本文件的完成状态。
4. 最后运行验证命令，并在回复里说明结果。

除非遇到下面情况，否则不再逐项询问：

- 需要改数据库结构且可能影响已有数据。
- 需要引入明显的新依赖。
- 需要改变产品定位或核心交互方向。
- 需要线上服务器操作、云资源调整或费用相关操作。

## 最终产品状态

项目最终不是单个StoryPlay作品的定制后台，而是一套可复用的StoryPlay平台。

最终要支持三类角色：

- 玩家：从前台大厅进入某个作品，获得接近传统StoryPlay的播放、选择和结局体验。
- 创作者：在后台创建项目、搭剧情树、上传素材、编辑片段、配置互动、试玩验证。
- 运营者：管理多个项目的首页展示、排序、隐藏、导入导出和发布检查。

最终形态包含：

- 前台项目大厅：展示多个StoryPlay 项目，每个项目有独立链接。
- 玩家端作品页：入口宣传页加正片播放页，选项叠在视频画面上。
- 后台编辑器：项目中心、剧情工作台、发布与迁移三层结构。
- 通用数据模型：项目、片段、选项、变量、时间线事件、试玩记录。
- 存储层：本地 SQLite 和远端 PostgreSQL 共享存储可切换。
- 导入导出：项目可迁移、可备份、可恢复。

## 产品原则

- 后台优先像编辑器，而不是普通表单后台。
- 高频创作动作集中在剧情工作台。
- 项目级配置和片段级编辑不要混在一起。
- 玩家能看到的内容要标清，玩家看不到的逻辑字段要弱化。
- 不为某个预设剧集硬编码字段，所有字段命名保持平台通用。
- 素材管理先满足上传和使用，不提前做过重资产库。
- 时间线和变量系统是平台差异点，要逐步从能用推进到好用。

## 信息架构

### 1. 前台大厅

职责：

- 展示多个项目。
- 每个项目展示标题、一句话介绍、封面或宣传视频。
- 每个项目提供独立访问链接。
- 提供“发布游戏”入口进入后台。

后续能力：

- 项目排序。
- 项目隐藏。
- 项目标签或分类。
- 项目状态展示。

### 2. 玩家端作品页

职责：

- 先展示入口宣传页。
- 正片以视频为核心。
- 选项直接叠在视频画面上。
- 视频结束出现选项时停留最后一帧。
- 支持倍速播放。
- 支持从指定片段试玩。
- 支持自动跳转、结尾选项、时间线中途互动。

后续能力：

- 更强结局页。
- 游玩进度恢复。
- 成就、图鉴、路线回看。

### 3. 后台项目中心

只处理项目级信息：

- 项目标题。
- 项目一句话介绍。
- 项目 slug。
- 首页是否展示。
- 首页排序。
- 入口宣传视频。
- 入口封面。
- 开始前说明。
- 起始片段。
- 全局变量。

不在这里编辑剧情树、片段内容或出口。

### 4. 后台剧情工作台

这是后台最高频界面。

职责：

- 创建片段。
- 查看剧情树。
- 从节点继续创建下一幕。
- 从节点创建分支。
- 编辑当前片段内容。
- 上传片段视频。
- 配置自动跳转。
- 配置结尾选项。
- 配置时间线事件。
- 从任意片段试玩。
- 标出缺视频、缺出口、无效目标、孤立片段。

目标布局：

- 左侧：项目切换、工作区切换、片段导航。
- 中间上方：剧情树画布。
- 中间下方：当前片段编辑器。
- 右侧：当前片段动作、继续创作、检查结果。

### 5. 发布与迁移

职责：

- 发布前检查。
- 导出项目 JSON。
- 导入项目 JSON。
- 重置为空白项目。
- 删除项目。
- 复制前台链接。
- 后续接入线上发布状态和版本记录。

## 技术分层

### 领域模型层

主要文件：

- `src/lib/story-engine.ts`

职责：

- 定义项目、片段、选项、变量、时间线事件结构。
- 保持模型通用，不写死具体作品字段。

### 存储抽象层

主要文件：

- `src/lib/storage.ts`
- `src/lib/sqlite.ts`
- `src/lib/postgres.ts`

职责：

- 根据环境选择 SQLite 或 PostgreSQL。
- 统一项目、片段、选项、试玩记录的读写。
- 承担旧库结构兼容和补列。

### 业务层

主要文件：

- `src/lib/game-store.ts`
- `src/lib/playthrough-store.ts`

职责：

- 校验输入。
- 校验节点存在性和编码唯一性。
- 清理删除片段后的引用。
- 处理导入导出数据。
- 管理试玩运行时状态。

### 管理接口层

主要文件：

- `src/app/api/admin/**`

职责：

- 给后台提供项目、片段、上传、导入导出接口。
- 返回明确错误信息。
- 不让前端直接接触存储细节。

### 玩家运行时接口层

主要文件：

- `src/app/api/playthroughs/**`

职责：

- 创建试玩。
- 推进片段。
- 处理玩家选择。
- 应用变量条件和动作。
- 支持从指定片段预览。

### 编辑器 UI 层

主要文件：

- `src/components/admin-story-editor.tsx`
- `src/components/branch-graph.tsx`

职责：

- 提供平台型后台编辑器。
- 组织项目中心和剧情工作台。
- 维护草稿状态。
- 调用后台接口保存。

### 玩家 UI 层

主要文件：

- `src/components/interactive-player.tsx`
- `src/app/page.tsx`
- `src/app/projects/[slug]/page.tsx`

职责：

- 提供前台大厅。
- 提供作品入口页。
- 提供正片播放、选项、倍速、结局体验。

## 当前完成状态

### 已完成能力

- 多项目模型。
- 首页项目大厅。
- 每个项目独立链接。
- 首页“发布游戏”入口。
- 后台选择项目。
- 项目创建、删除、导入、导出。
- 宣传视频上传。
- 片段视频上传。
- SQLite / PostgreSQL 双存储模式。
- 变量模型。
- 时间线事件模型。
- 条件和动作模型。
- 前台时间线事件消费。
- 前台视频选项叠层。
- 前台倍速播放。
- 前台视频结束后保留最后一帧。
- 后台项目中心和剧情工作台初步合并。
- 剧情树节点状态展示。
- 剧情树节点快捷动作：编辑、试玩、加下一幕、加分支。

### 当前主要问题

- 后台功能已经多，但编辑心智仍不够清晰。
- 剧情树还是静态布局，不是真正画布。
- 片段编辑仍偏表单，视频预览和出口编辑不够紧凑。
- 时间线编辑器还不够可视化。
- 发布检查没有形成独立闭环。
- 素材系统仍是 URL 回填模式。

## 开发阶段

### Phase 1：平台基础闭环

状态：已完成。

目标：

- 能创建项目。
- 能创建片段。
- 能上传素材。
- 能配置入口页。
- 能搭基础分支。
- 能前台播放。
- 能导入导出。

验收标准：

- 新项目可以从空项目开始配置。
- 至少一个项目可以在首页展示并进入正片。
- 后台保存后刷新不丢数据。
- 导出后再导入可恢复项目。
- `npm run lint` 和 `npm run build` 通过。

### Phase 2：后台主工作流收敛

状态：已完成第一轮，继续维护。

目标：

- 后台不再像散乱表单。
- 项目级配置和片段级创作分开。
- 剧情树和片段编辑放回同一个工作台。

已完成：

- 工作区收敛为“项目”和“剧情”。
- 剧情工作台顶部状态条展示片段、出口、时间线事件和结构状态。
- 当前片段编辑按“片段内容 -> 视频与流转 -> 结尾出口 -> 时间线”组织。
- 右侧保留单一创作动作面板。
- 当前片段可直接创建并连接新片段。

后续维护项：

- 新项目、空项目、无片段状态继续简化。
- 继续清理重复字段和出戏文案。
- 持续标注玩家可见字段和逻辑字段。

验收标准：

- 用户进入后台能明确知道先做项目配置，再做剧情。
- 新增片段、编辑片段、继续创作入口清晰。
- 页面中不出现预设剧集字段或调试型文案。

### Phase 3：剧情树画布化

状态：当前阶段。

目标：

- 剧情树成为主要创作入口，而不是只读结构图。
- 创作者能从节点直接继续创作、分支、试玩和定位问题。

已完成：

- 节点卡片显示视频、出口、时间线状态。
- 节点卡片提供编辑、试玩、加下一幕、加分支。
- 问题节点用颜色区分。
- 加下一幕和加分支会预填右侧继续创作表单，不直接创建空节点。

下一步任务：

- 增加剧情树筛选：全部、问题、起始、结局、孤立。
- 增加画布工具条：适应宽度、重置视图、只看问题。
- 增加节点问题摘要：缺视频、缺出口、无效目标分开显示。
- 增加结构视图切换：剧情树、问题列表、片段列表。
- 优化空项目和首个片段创建路径。

暂不做：

- 拖拽改线。
- 节点自由拖动保存布局。
- 小地图。
- 复杂缩放。

验收标准：

- 用户能在剧情树上完成大部分结构搭建入口操作。
- 用户能一眼看到哪个节点有问题。
- 用户能从问题节点直接进入修复。
- 不引入新的数据库结构。
- `npm run lint` 和 `npm run build` 通过。

### Phase 4：片段编辑器产品化

状态：未开始。

目标：

- 片段编辑从“大表单”升级成“创作面板”。
- 视频、内容、出口、时间线围绕当前片段形成清晰编辑体验。

任务：

- 把片段视频预览放进编辑器。
- 视频上传后可直接预览。
- 片段顶部工具条固定：保存、试玩、复制、删除、设为起点。
- 出口选项改为紧凑卡片列表。
- 自动跳转和选项出口互斥关系更清楚。
- 结局片段隐藏无意义字段。
- 未保存状态更明显。

验收标准：

- 编辑一个片段不需要在多个区域来回找入口。
- 上传视频后能立即看到预览。
- 配置出口时不再像填大表。
- 结局片段不会展示无关的跳转字段。

### Phase 5：时间线编辑器可视化

状态：未开始。

目标：

- 把中途互动从配置表单升级成时间轴操作。

任务：

- 用时间轴标尺展示事件分布。
- 事件按轨道分类：文本、选项、叠层、跳转、动作。
- 每个事件用类型化表单编辑，不直接暴露 JSON。
- 支持按当前视频时间插入事件。
- 支持复制事件。
- 支持启用、禁用事件。
- 支持按时间排序和快速微调。

验收标准：

- 用户能理解某个事件发生在视频哪个时间点。
- 高频事件不用手写 JSON。
- 同一个片段里多个中途互动能清楚管理。

### Phase 6：发布检查与运营化

状态：未开始。

目标：

- 作品上线前能集中检查问题。
- 多项目运营更可控。

任务：

- 发布前检查独立成面板。
- 检查起始片段。
- 检查缺视频。
- 检查缺出口。
- 检查无效目标。
- 检查未展示项目和缺宣传素材。
- 提供前台链接复制。
- 后续支持发布状态和版本记录。

验收标准：

- 发布前能看到明确的可修复问题列表。
- 每个问题都能跳转到对应位置。
- 项目链接可一键复制。

### Phase 7：素材管理轻量化

状态：延后。

目标：

- 先满足真实上传和复用，不做复杂资产库。

任务：

- 后台显示最近上传素材。
- 支持从已上传素材选择。
- 项目入口素材和片段视频统一上传体验。
- 后续再做标签、搜索、批量管理。

验收标准：

- 不需要重复上传同一个视频。
- 可以知道素材被哪个项目或片段使用。

## 当前执行队列

### IA 修正：前台大厅和后台项目入口

目标：

- 前台首页不是普通项目卡片列表，而是基于作品宣传页延展成“纵向宣传片流”。
- 向下滚动就是其他项目的宣传片和入口信息。
- 后台 `/admin` 先进入项目总览页。
- 从项目总览页进入具体项目编辑页 `/admin?project=slug`。

验收标准：

- 前台首页每个项目是一个完整宣传段落，优先展示宣传视频，其次展示封面。
- 首页保留“发布游戏”入口。
- 后台无项目参数时展示项目总览，不直接进入某个项目编辑器。
- 后台项目总览能进入项目编辑、前台预览，并能新建项目。
- 不改数据库结构。
- `npm run lint` 和 `npm run build` 通过。

- 问题列表和剧情树联动。

### P1

### P2

- 时间线事件轨道雏形。
