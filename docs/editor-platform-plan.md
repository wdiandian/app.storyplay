# 互动影游平台编辑器开发计划

## 目标调整

当前项目不再继续朝“单个作品的定制后台”扩展，而是转向“可复用的互动影游平台编辑器”。

这轮改造的核心不是继续追加某个具体作品字段，而是把后台抽象成一套通用内容生产工具，支持：

- 创建和管理项目
- 管理宣传素材与正片素材
- 构建场景、分支、跳转关系
- 配置项目变量与运行时状态
- 配置场景时间线事件
- 导入导出项目数据
- 本地或远端共享数据库运行

## 当前代码现状

### 已完成

- 基础项目 / 场景 / 选项的数据模型已存在
- 前后端已有基础互动播放能力
- 后台已支持：
  - 视频上传到服务器本地目录
  - 项目导入
  - 项目导出
  - 宣传视频字段
- 存储层已支持 SQLite 和 PostgreSQL 双模式切换
- 后台主界面已开始从“表单页”重构成“编辑器工作台”

### 已落地的平台化能力

- `StoryGame.variables`
- `StoryNode.timelineEvents`
- SQLite 对应 JSON 字段
- PostgreSQL 对应 JSON 字段与旧表补列
- 管理后台中的变量编辑 UI
- 管理后台中的时间线事件编辑 UI
- 前台运行时对时间线事件的消费

## 产品方向

## 一期目标：编辑器骨架可用

先交付一版“平台型编辑器骨架”，优先保证可编辑、可保存、可导入导出、可运行，不追求一次做成完整可视化工业工具。

一期交付标准：

- 可以创建一个空项目
- 可以上传宣传视频、宣传封面、场景视频
- 可以创建场景
- 可以配置起始场景
- 可以编辑场景基础信息
- 可以配置场景分支与自动跳转
- 可以配置项目变量
- 可以配置场景时间线事件
- 可以导出项目 JSON
- 可以重新导入同一份 JSON
- 前台仍可正常播放现有项目

## 编辑器信息架构

后台编辑器先收敛为 5 个工作区：

### 1. Project

管理项目级配置：

- 项目标题
- 副标题 / 一句话介绍
- 项目简介
- 宣传页标题
- 宣传页文案
- 宣传视频 URL
- 宣传封面 URL
- 起始场景
- 项目变量

### 2. Assets

管理素材资源：

- 宣传视频
- 宣传封面
- 场景视频
- 后续扩展：角色立绘、背景图、音频、字幕文件

一期仍先使用“上传后拿到 URL”的轻量素材方案，不先做复杂资产引用系统。

### 3. Flow

展示剧情流转结构：

- 当前场景关系图
- 起始场景标记
- 指向不存在节点的选项提示
- 无出口非结局场景提示

一期先保留现有 branch graph，后续再升级为可拖拽画布。

### 4. Scene

管理单个场景：

- 标题
- 描述
- 台词 / 文本稿
- 视频地址
- 场景类型
- 自动跳转
- 结局类型
- 时间线事件

### 5. Choices

管理场景出口：

- 选项文案
- 选项编码
- 选项提示
- 目标场景

后续扩展：

- 条件显示
- 变量修改
- 倒计时选项
- 权重随机

## 数据模型规划

### Project Variables

项目级变量用于承载通用互动逻辑，而不是继续写死某个作品字段。

```ts
type VariableDefinition = {
  key: string;
  label: string;
  type: "number" | "boolean" | "string" | "enum";
  initialValue: string | number | boolean;
  options?: string[];
};
```

用途示例：

- 好感度
- 阵营
- 线索是否获得
- 玩家身份状态
- 任意平台化业务变量

### Timeline Events

时间线事件用于描述“某个场景在某个时间点要发生什么”。

```ts
type TimelineEvent = {
  id: string;
  atMs: number;
  type:
    | "show_text"
    | "show_choice"
    | "pause"
    | "play_audio"
    | "jump"
    | "run_actions"
    | "show_overlay";
  payload: Record<string, unknown>;
};
```

用途示例：

- 视频中途弹字幕
- 视频中途弹选项
- 到某个时间点暂停
- 播放音效
- 直接跳场景
- 执行动作
- 显示 UI 浮层

这是把“互动影游平台”拉开差异的关键，不应继续停留在“视频播完才给选项”的单一模式。

## 技术分层

### 1. 领域模型层

文件：

- `src/lib/story-engine.ts`

职责：

- 定义项目、场景、选项、变量、时间线事件结构
- 提供基础查询工具

### 2. 存储抽象层

文件：

- `src/lib/storage.ts`
- `src/lib/sqlite.ts`
- `src/lib/postgres.ts`

职责：

- 根据环境决定走 SQLite 或 PostgreSQL
- 统一提供项目、场景、选项、试玩记录的存取能力
- 承担兼容旧库结构的迁移职责

### 3. 业务层

文件：

- `src/lib/game-store.ts`

职责：

- 校验用户输入
- 节点存在性校验
- 编码唯一性校验
- 导入数据清洗
- 导入数据结构校验

### 4. 管理接口层

文件：

- `src/app/api/admin/**`

职责：

- 提供后台管理接口
- 返回明确错误信息
- 避免前端直接接触存储细节

### 5. 编辑器 UI 层

文件：

- `src/components/admin-story-editor.tsx`

职责：

- 提供平台型后台编辑器
- 组织工作区
- 维护草稿状态
- 调用后台接口保存

## 分阶段实施计划

## Phase 1: 平台基础能力闭环

目标：先把“变量 + 时间线事件 + 编辑器骨架”做成一条完整链路。

任务：

- 存储层补齐 `variables` / `timelineEvents`
- SQLite / PostgreSQL 兼容旧结构
- 完成 `game-store.ts` 的变量和事件校验
- 完成后台 API 参数接入
- 完成编辑器 Project 面板变量 UI
- 完成编辑器 Scene 面板时间线事件 UI
- 跑通 `lint` 和 `build`

验收：

- 本地创建变量后可保存并刷新不丢
- 场景时间线事件可保存并刷新不丢
- 导出 JSON 包含变量和时间线事件
- 导入 JSON 后数据恢复正常

## Phase 2: 后台从表单升级为工作台

目标：让后台结构真正像“内容生产工具”。

任务：

- 规范左侧导航与工作区切换
- 明确项目、素材、流程、场景、选项分区
- 清理作品定制化字段和文案
- 修复现有组件中的乱码文本
- 统一中文界面语言

验收：

- 后台信息架构清晰
- 不再依赖某个预设剧集字段
- 新项目从基础信息开始创建，而不是默认有剧情树

## Phase 3: 时间线驱动的互动能力

目标：从“播完再选”升级为“场景内实时互动”。

任务：

- 播放器接入 `timelineEvents`
- 支持在视频时间点显示选项
- 支持时间点弹文本 / 浮层
- 支持时间点暂停
- 支持事件触发后保留最后一帧

验收：

- 同一场景可在播放中途出现交互
- 事件型交互不依赖视频结束统一弹出

## Phase 4: 条件与动作系统

目标：让分支逻辑从静态跳转升级为可计算逻辑。

任务：

- 给选项加条件表达式
- 给选项和事件加变量修改动作
- 增加运行时变量状态
- 存储游玩过程中的变量变化

验收：

- 同一场景的可见选项可因变量不同而变化
- 玩家选择可改变后续路线

## Phase 5: 资源系统升级

目标：把后台从“填 URL”推进到“资产可管理”。

任务：

- 建立 asset 实体
- 上传后返回 `assetId`
- 场景、宣传页、角色图改为引用资产
- 增加角色立绘、背景图、音频、字幕支持

验收：

- 素材可重复引用
- 项目层和场景层素材关系清晰

## 当前实现优先级

下一轮继续按下面顺序推进：

1. 清理后台编辑器乱码文案
2. 扩展时间线事件专用表单
3. 把流程图升级为更接近画布编辑器的结构
4. 引入条件 / 动作模型
5. 继续抽象素材系统

## 已知风险

### 1. 展示层仍有乱码残留

当前主要集中在：

- `src/components/admin-story-editor.tsx`

这部分不影响数据结构和构建，但会影响编辑器可用性和验收体验，需要继续清理。

### 2. 时间线编辑器仍是混合模式

当前已对高频事件提供部分专用字段，但仍保留 `payload JSON` 兜底。

这是刻意保留的折中：

- 优点是扩展快
- 缺点是编辑体验还不够“完整编辑器”

### 3. 素材系统仍是轻量 URL 模式

现在的上传功能本质还是服务器本地文件 + URL 回填，不是完整资产管理系统。

## 完成定义

本轮“编辑器平台化改造”完成的标准不是页面更好看，而是下面这些都成立：

- 后台字段命名通用
- 后台围绕“项目创建与编辑”组织，而不是围绕预设剧情
- 数据模型支持变量和时间线事件
- SQLite / PostgreSQL 都能保存这些能力
- 导入导出完整
- 构建通过
- 前后台主流程可运行
