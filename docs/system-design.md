# 系统设计

本文档描述 StoryPlay App 的目标系统形态。它用于指导后续重构，不代表当前代码已经全部实现。

## 1. 目标

StoryPlay App 要稳定支撑一个完整的手工分支影游闭环：

1. 创作者在创作后台编辑游戏。
2. 创作者通过发布检查后发布游戏。
3. 首页展示可公开游玩的游戏。
4. 玩家进入播放页游玩发布版本。
5. 系统记录一次游玩过程，并根据节点、选择、条件、变量和时间线事件推进剧情。

项目边界：

- 本项目负责手工分支影游和创作后台。
- 主站负责 AI 实时生成互动影游。
- 本项目不与主站仓库合并。

## 2. 产品信息架构

### 2.1 首页

路径：

- `/`

定位：

- 玩家发现游戏的入口。
- 只展示可被玩家进入的项目。
- 不承担复杂编辑、配置、模型设置或营销说明。

核心模块：

- 顶部品牌与轻量导航。
- 推荐/精选游戏区域。
- 游戏卡片或全屏作品展示。
- 游戏基本信息：标题、简介、封面、标签、分类、更新时间。
- 进入播放页按钮。
- 弱化的创作后台入口。

需要避免：

- 直接暴露草稿项目。
- 加载完整游戏节点图。
- 把创作后台说明写成首页主要内容。

### 2.2 播放页

路径：

- `/projects/[slug]`

定位：

- 玩家实际游玩单个游戏的运行时界面。

核心模块：

- 游戏进入状态。
- 宣传/片头状态。
- 视频节点播放。
- 分支选择。
- 自动跳转提示。
- 时间线事件展示。
- 信息面板。
- 结局展示。
- 重新开始。
- 错误和空状态。

播放页应该关心：

- 当前播放的是哪个项目。
- 当前 playthrough 的状态。
- 当前节点是什么。
- 当前可用选择是什么。
- 当前变量和已触发事件是什么。

播放页不应该关心：

- 后台编辑表单如何组织。
- 项目是否正在被创作者编辑。
- 首页如何排序。

### 2.3 创作后台

路径：

- `/admin`
- 目标路径：`/admin/projects/[slug]`
- 兼容路径：`/admin?project=[slug]`

定位：

- 创作者管理和编辑游戏的工作台。

核心工作区：

- 项目总览。
- 项目设置。
- 展示信息。
- 节点目录。
- 节点编辑。
- 分支编辑。
- 时间线事件。
- 变量、条件和动作。
- 分支图。
- 发布检查。
- 导入导出。
- 素材上传。

后台应该优先保证：

- 结构清晰。
- 错误可发现。
- 编辑不容易误操作。
- 保存和预览路径明确。
- 发布前能知道项目是否可玩。

## 3. 数据模型设计

### 3.1 Project

Project 是项目级实体，代表一个创作者维护的作品。

职责：

- 标识项目。
- 保存展示信息。
- 保存编辑状态。
- 后续承载发布状态、权限、版本信息。

当前对应：

- `StoryGame` 中的一部分字段。

建议后续明确字段：

- `id`
- `slug`
- `title`
- `tagline`
- `description`
- `coverUrl`
- `promoVideoUrl`
- `promoPosterUrl`
- `tags`
- `category`
- `visibility`
- `publishStatus`
- `createdAt`
- `updatedAt`
- `publishedAt`

### 3.2 Game

Game 是可运行的剧情图结构。

职责：

- 定义起始节点。
- 定义节点集合。
- 定义全局变量。
- 定义运行规则。

当前对应：

- `StoryGame`

核心字段：

- `startNodeCode`
- `variables`
- `nodes`

后续需要区分：

- Draft Game：创作者正在编辑的草稿。
- Published Game：玩家实际游玩的发布版本。

### 3.3 Node

Node 是游戏中的一个剧情片段。

职责：

- 保存片段标题、描述、正文、视频地址。
- 标识节点类型。
- 保存默认选择。
- 保存自动跳转。
- 保存时间线事件。
- 标识结局。

当前对应：

- `StoryNode`

核心字段：

- `code`
- `title`
- `description`
- `transcript`
- `videoUrl`
- `nodeType`
- `autoNextNodeCode`
- `choices`
- `timelineEvents`
- `isEnding`
- `endingTone`

### 3.4 Choice

Choice 是玩家或时间线事件提供的分支选项。

职责：

- 显示选项文案。
- 指向目标节点。
- 根据条件决定是否可用。
- 执行动作修改变量。

当前对应：

- `StoryChoice`

核心字段：

- `code`
- `label`
- `hint`
- `targetNodeCode`
- `conditions`
- `actions`

### 3.5 TimelineEvent

TimelineEvent 是节点播放过程中的时间线事件。

职责：

- 在指定时间触发 UI、跳转、动作或提示。
- 支持条件判断。
- 支持触发后修改变量。

当前对应：

- `TimelineEvent`

当前事件类型：

- `show_text`
- `show_choice`
- `pause`
- `play_audio`
- `jump`
- `run_actions`
- `show_overlay`

后续目标：

- 为每种事件类型定义明确 payload schema。
- 在后台提供可视化编辑，而不是主要依赖 JSON。

### 3.6 Variable / Condition / Action

Variable 是游戏运行时状态。

Condition 用于判断某个选择或事件是否可用。

Action 用于改变运行时变量。

当前对应：

- `VariableDefinition`
- `ConditionRule`
- `VariableAction`
- `VariableState`

后续目标：

- 后台编辑器提供更清楚的变量面板。
- 发布检查能发现不存在的变量引用。
- 播放页只执行规则，不负责解释编辑器结构。

### 3.7 Playthrough

Playthrough 是一次玩家游玩记录。

职责：

- 记录当前节点。
- 记录已做选择。
- 记录变量状态。
- 记录已触发事件。
- 标识是否完成。

当前对应：

- `PlaythroughState`

核心字段：

- `id`
- `gameSlug`
- `currentNodeCode`
- `status`
- `history`
- `startedAt`
- `finishedAt`
- `variables`
- `triggeredEventIds`

后续需要补充：

- 所属发布版本。
- 玩家身份。
- 运行时快照。

## 4. 首页设计

首页只消费 ProjectSummary 或类似摘要数据。

当前最小展示逻辑：

- 读取所有项目摘要。
- 过滤 `listedOnHome`。
- 按 `sortOrder` 和 `updatedAt` 排序。
- 展示标题、简介、海报/宣传视频。
- 进入 `/projects/[slug]`。

后续产品化方向：

- 只展示 `publishStatus = published` 且 `visibility = public` 的项目。
- 支持分类和标签。
- 支持精选排序。
- 支持空状态和运营占位。
- 支持播放量、更新时间、作品状态等辅助信息。

首页不应直接依赖：

- `StoryNode`
- `StoryChoice`
- `TimelineEvent`
- 编辑器草稿状态。

## 5. 播放页设计

播放页的核心状态机：

1. 加载项目。
2. 创建 playthrough。
3. 显示宣传/片头。
4. 播放当前节点视频。
5. 触发时间线事件。
6. 节点结束后显示可用选择，或自动跳转。
7. 更新 playthrough。
8. 进入下一个节点。
9. 如果到达结局，展示结局面板。

关键异常状态：

- 项目不存在。
- 项目没有起始节点。
- 当前节点不存在。
- 当前节点缺少视频。
- 当前节点没有选择，也没有自动跳转，也不是结局。
- playthrough 不存在。
- 时间线事件 payload 不合法。

播放页重构目标：

- 把 API 请求和 payload 类型移出 UI 主组件。
- 把 timeline payload 解析移出 UI 主组件。
- 把 scene presentation 常量移出 UI 主组件。
- 把选择面板、信息面板、结局面板拆成子组件。
- 保持视频控制和主运行时状态稳定，后续再拆。

## 6. 创作后台设计

### 6.1 项目总览

职责：

- 展示所有项目。
- 新建项目。
- 进入项目编辑。
- 进入前台预览。
- 显示首页展示状态、排序和最近更新时间。

后续增强：

- 显示草稿/已发布状态。
- 显示发布检查结果。
- 支持复制项目、删除项目、归档项目。

### 6.2 项目设置

职责：

- 编辑标题、slug、简介。
- 编辑首页展示信息。
- 编辑宣传视频和海报。
- 设置是否首页展示。
- 设置排序。
- 设置起始节点。
- 编辑全局变量。

### 6.3 节点编辑

职责：

- 创建节点。
- 编辑节点标题、描述、正文、视频地址。
- 设置节点类型。
- 设置自动跳转。
- 设置结局类型。
- 删除节点并清理引用。

### 6.4 分支编辑

职责：

- 编辑默认选择。
- 设置目标节点。
- 设置选择条件。
- 设置选择动作。
- 检查重复 code 和无效目标。

### 6.5 时间线事件

职责：

- 在指定时间显示文本、选项、叠层或执行动作。
- 支持暂停、跳转和音频事件。
- 支持条件和动作。

当前问题：

- payload 仍偏 JSON 化。

后续目标：

- 根据事件类型提供结构化表单。
- 通过校验保证 payload 可运行。

### 6.6 变量、条件和动作

职责：

- 定义全局变量。
- 在选择和事件中引用变量。
- 通过条件控制可见性。
- 通过动作更新运行时状态。

后续目标：

- 变量被引用时不允许随意删除。
- 发布检查能列出不存在或类型不匹配的变量引用。

### 6.7 发布检查

职责：

- 判断项目是否可被玩家稳定游玩。
- 把结构问题展示给创作者。
- 阻止明显不可玩的项目发布。

首批检查项：

- 起始节点存在。
- 每个节点 code 唯一。
- 每个选择目标存在。
- 每个自动跳转目标存在。
- 时间线跳转目标存在。
- 时间线选择目标存在。
- 条件变量存在。
- 动作变量存在。
- 非结局节点至少有选择或自动跳转，除非明确允许停留。
- 结局节点配置完整。

## 7. 发布流设计

当前项目只有 `listedOnHome`，还不是完整发布流。

目标发布流：

1. 创作者编辑草稿。
2. 系统运行发布检查。
3. 检查通过后生成发布版本。
4. 首页只展示已发布且公开的版本。
5. 玩家游玩发布版本。
6. 创作者继续编辑草稿时，不影响正在游玩的发布版本。

建议状态：

- `draft`：草稿。
- `ready`：检查通过但未发布。
- `published`：已发布。
- `archived`：归档。

建议可见性：

- `private`：仅后台可见。
- `unlisted`：有链接可访问，但不在首页展示。
- `public`：首页可展示。

发布版本需要解决：

- 版本 ID。
- 发布时间。
- 发布快照。
- 玩家 playthrough 绑定版本。
- 新版本发布后旧 playthrough 的处理策略。

## 8. 技术模块设计

目标目录方向：

```text
src/
  app/
  components/
    home/
    player/
    admin/
  lib/
    domain/
    services/
    runtime/
    storage/
    admin/
```

### 8.1 domain

职责：

- 类型定义。
- 图结构分析。
- 发布检查。
- 条件和动作规则。
- payload schema 校验。

### 8.2 services

职责：

- 项目服务。
- 游玩服务。
- 发布服务。
- 上传服务。

服务层不应该直接包含 UI 状态。

### 8.3 runtime

职责：

- 根据当前 playthrough 和当前节点计算下一步。
- 处理选择、自动跳转、时间线事件。
- 保证运行时逻辑可测试。

### 8.4 storage

职责：

- 统一 SQLite/PostgreSQL 接口。
- 隐藏具体存储实现。
- 保持旧数据兼容。

### 8.5 components

职责：

- `components/home`：首页展示组件。
- `components/player`：播放页运行时组件。
- `components/admin`：创作后台组件。

组件层不应承担过多领域校验和数据转换。

## 9. 阶段实施建议

### 阶段 1：文档和结构边界

- 完成中文文档体系。
- 完成代码盘点。
- 明确重构执行清单。
- 不改运行行为。

### 阶段 2：后台编辑器拆分

- 先拆 helper 和常量。
- 再拆小 UI 组件。
- 最后拆工作区面板。
- 每步保持页面行为不变。

### 阶段 3：播放器拆分

- 先拆 payload 类型和 helper。
- 再拆展示子组件。
- 最后抽运行时状态逻辑。

### 阶段 4：领域校验和发布检查

- 建立 `domain` 目录。
- 抽出图结构检查。
- 在后台展示发布检查结果。

### 阶段 5：发布流和首页收敛

- 增加发布状态和可见性。
- 首页只读可公开的发布内容。
- 播放页绑定发布版本。

## 10. 当前不做

- 不做账号和权限系统。
- 不做收费和额度系统。
- 不做大规模 UI 重设计。
- 不做破坏性数据库迁移。
- 不合并主站代码。
