# UI 设计规范

本文档定义 StoryPlay App 后续界面重构的视觉方向和基础规则。它不是当前界面的复刻，也不是某个现成产品规范的照搬，而是给首页、播放页和创作后台建立统一但分工明确的设计语言。

参考方向：

- 播放页参考 Runway 一类影像工具的沉浸式暗色视频界面。
- 首页参考 PlayStation 的作品入口感，以及 Pinterest 的内容发现效率。
- 创作后台参考 Sanity 的内容结构感，以及 Airtable 的密集数据编辑体验。
- 以上参考只用于判断信息密度、色彩性格、组件组织和交互节奏，不复制品牌视觉。

## 1. 设计原则

### 1.1 三个界面三种任务

StoryPlay App 不应该所有页面都长得一样。不同模块的任务差异很大：

- 首页：让玩家快速发现可玩的作品。
- 播放页：让玩家沉浸在一个具体故事里。
- 创作后台：让创作者稳定、清晰、低误操作地编辑复杂分支内容。

统一的是品牌气质、基础组件、排版规则和状态表达；差异化的是页面密度、背景明暗、交互节奏和视觉重心。

### 1.2 影像优先

这个项目的核心内容是互动影游，视觉设计必须让视频、封面、海报和节点内容成为主角。

优先级：

1. 影像内容。
2. 当前任务。
3. 可执行操作。
4. 辅助说明。

首页和播放页不要用大量解释性文字替代作品本身。后台可以承载更多文本和表单，但也要让结构关系优先于装饰。

### 1.3 工具界面保持克制

创作后台是工作台，不是营销页。

后台应避免：

- 过大的宣传式标题。
- 装饰性渐变背景。
- 多层卡片嵌套。
- 大面积空白导致信息扫描效率低。
- 把操作说明写成页面主体内容。

后台应优先：

- 稳定布局。
- 清楚分组。
- 明确保存、预览、发布状态。
- 可见的校验反馈。
- 高效处理节点、选择、变量和时间线事件。

## 2. 首页设计

首页定位为“影游发现页”，不是产品介绍页。

### 2.1 视觉方向

首页应当有作品入口感，第一屏就能看到具体游戏或精选内容。可以使用大幅封面、视频海报、横向精选区、作品栅格，但不要做只介绍平台能力的落地页。

建议气质：

- 影像化。
- 内容驱动。
- 有游戏入口感。
- 信息简洁但可扫描。

不建议：

- 用抽象渐变或插画占据首屏。
- 首屏只有口号，没有具体作品。
- 大量解释“如何使用这个网站”。
- 所有作品卡片都套厚重阴影和大圆角。

### 2.2 布局

首页建议包含：

- 顶部品牌和轻量导航。
- 精选作品区。
- 作品列表或分类区域。
- 游戏卡片。
- 进入播放页的明确操作。
- 弱化但可找到的创作后台入口。

作品卡片必须稳定显示：

- 封面或海报。
- 标题。
- 简短简介。
- 标签或类型。
- 状态或更新时间。
- 进入按钮。

### 2.3 信息密度

首页的信息密度应介于播放页和后台之间：

- 比播放页更适合浏览和比较。
- 比后台更视觉化、更少表单感。
- 移动端优先保证作品卡片可读和可点。

## 3. 播放页设计

播放页定位为“沉浸式互动影像运行时”。

### 3.1 视觉方向

播放页应以暗色、全屏视频、低干扰控制为主。玩家进入故事后，界面要像一个正在运行的互动影片，而不是表单页或管理页。

建议气质：

- 暗色。
- 电影感。
- 控制克制。
- 选择明确。
- 状态反馈及时。

不建议：

- 播放器周围堆叠大量卡片。
- 使用明亮后台式表单控件。
- 在剧情播放区显示大量解释文案。
- 让调试信息、编辑信息或后台状态进入玩家界面。

### 3.2 播放状态

播放页至少要覆盖这些状态：

- 加载中。
- 宣传/片头。
- 视频播放中。
- 时间线事件触发。
- 分支选择。
- 自动跳转。
- 结局。
- 错误。
- 重新开始。

每个状态都应有明确的视觉优先级。视频播放中，视频最大；选择出现时，选择最大；错误发生时，修复路径最大。

### 3.3 分支选择

选择按钮不是普通表单按钮，而是剧情决策入口。

规则：

- 选择文案要足够可读。
- 目标动作要明确。
- 不可用选择需要说明原因，不能只变灰。
- 多个选择之间要有清楚层级。
- 移动端选择区不能遮挡关键视频内容太久。

### 3.4 播放控制

播放控制应保持低干扰：

- 使用图标按钮表达播放、暂停、重播、静音、全屏。
- 对不常用操作使用悬浮层或次级区域。
- 重要剧情选择不能和播放器控制混在同一层级。
- 时间线事件提示应短、准、可消失。

## 4. 创作后台设计

创作后台定位为“结构化影游创作工具”。

### 4.1 视觉方向

后台要稳定、紧凑、可扫描。它的设计目标不是让页面显得华丽，而是让复杂分支结构可以被创作者长期维护。

建议气质：

- 明亮或中性底色。
- 高可读文字。
- 清楚边界。
- 稳定布局。
- 低装饰。
- 高信息密度。

不建议：

- 大面积暗色电影风。
- 营销式大标题。
- 多层嵌套卡片。
- 大面积渐变或装饰背景。
- 复杂操作被隐藏到难以发现的位置。

### 4.2 工作区结构

后台建议采用固定工作区模型：

- 左侧：项目、节点或章节目录。
- 中间：当前编辑主体。
- 右侧：属性、校验、预览或上下文信息。
- 顶部：保存、预览、发布、导入导出等全局动作。

当屏幕宽度不足时：

- 左侧目录可折叠。
- 右侧上下文面板可抽屉化。
- 当前编辑主体优先保留。
- 关键操作不能只存在于被折叠区域。

### 4.3 表单与编辑

后台表单规则：

- 字段标签清楚、短。
- 同类字段归组。
- 危险操作放在低频区域，并需要确认。
- 保存状态要可见。
- 校验错误靠近字段显示。
- 发布检查错误要能定位到对应节点、选择或事件。

复杂配置要逐步结构化：

- 时间线事件不要长期依赖裸 JSON。
- 条件和动作应使用可组合控件。
- 变量引用应提供选择器。
- 节点目标应提供节点选择，不只输入文本。

## 5. 色彩系统

StoryPlay App 不采用单一紫蓝渐变作为主视觉。色彩需要同时服务影像、发现和创作工具三类场景。

### 5.1 基础色

建议基础色：

- 页面底色：`#f7f4ef` 或接近的暖白，用于首页和后台。
- 后台表面：`#ffffff`、`#f3f1ec`、`#ebe7df`。
- 播放页底色：`#08090b`、`#111318`、`#1a1d24`。
- 主要文字：`#161616`。
- 暗色文字：`#f5f2ea`。
- 次级文字：`#6f6a60`。
- 暗色次级文字：`#a8a29a`。

### 5.2 品牌强调色

建议使用偏电影和互动感的暖红作为主强调色：

- 主强调：`#e0442e`。
- 主强调 hover：`#c93422`。
- 低强调背景：`#fff0eb`。

辅助色只用于状态和分类，不抢主视觉：

- 成功：`#2f8f5b`。
- 警告：`#b7791f`。
- 错误：`#c53030`。
- 信息：`#2b6cb0`。

### 5.3 使用规则

- 首页可以使用封面图本身形成色彩丰富度。
- 播放页主要靠影像内容和少量强调色。
- 后台颜色主要用于状态、选中、校验和操作层级。
- 不要把每个模块都做成不同颜色的装饰卡片。

## 6. 字体与排版

### 6.1 字体

优先使用系统字体栈：

```css
font-family: Inter, ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
```

中文环境下保持系统默认中文字体渲染，不引入不必要的远程字体。

### 6.2 字号

建议字号：

- 页面标题：28-40px，仅首页和关键页面使用。
- 区块标题：20-24px。
- 面板标题：16-18px。
- 正文：14-16px。
- 表单和表格：13-14px。
- 辅助信息：12-13px。

后台不要滥用大字号。播放页可以在片头、结局和关键选择上使用更强的标题层级。

### 6.3 排版规则

- 字间距保持默认，不使用负字距。
- 按钮和卡片内文字必须完整可读。
- 长标题需要换行或截断，并提供可查看完整内容的方式。
- 不用视口宽度动态缩放字体。

## 7. 间距与布局

### 7.1 基础间距

使用 4px 或 8px 的间距节奏：

- 4px：图标和文字之间。
- 8px：紧密字段间距。
- 12px：表单项内间距。
- 16px：常规组件间距。
- 24px：区块间距。
- 32px：页面级分隔。

### 7.2 圆角

- 工具按钮：6-8px。
- 输入框：6-8px。
- 卡片：8px 以内。
- 播放页叠层：8-12px。

不要让所有界面都变成大圆角、厚阴影、漂浮卡片风格。

### 7.3 阴影

阴影只用于表达层级：

- 弹窗。
- 抽屉。
- 悬浮菜单。
- 播放页叠层。

后台普通面板优先用边框和背景区分，不依赖重阴影。

## 8. 组件规则

### 8.1 按钮

按钮层级：

- 主按钮：当前页面最重要动作。
- 次按钮：可选动作。
- 文本按钮：低频或轻量动作。
- 图标按钮：播放、关闭、折叠、上传、删除、复制等明确工具动作。

规则：

- 常见工具操作优先使用图标加 tooltip。
- 危险操作使用红色或警示状态，并需要确认。
- 同一区域不要出现多个同级主按钮。

### 8.2 卡片

卡片只用于：

- 首页作品项。
- 后台列表项。
- 后台重复实体。
- 弹窗或独立工具面板。

不要把页面大区块全部做成卡片，也不要卡片套卡片。

### 8.3 输入控件

根据数据类型选择控件：

- 布尔值使用开关或复选框。
- 枚举使用下拉、分段控件或单选。
- 数字使用输入框、步进器或滑杆。
- 颜色使用色块。
- 节点目标使用选择器。
- 条件和动作使用结构化编辑器。

### 8.4 状态反馈

必须覆盖：

- 加载中。
- 空状态。
- 保存中。
- 已保存。
- 保存失败。
- 校验错误。
- 发布检查失败。
- 网络错误。

后台错误提示要指向可修复的位置。播放页错误提示要给玩家可继续的路径，例如重试、返回首页、重新开始。

## 9. 动效与交互

动效服务状态变化，不做装饰。

建议：

- 首页作品 hover 可轻微放大封面或显示进入操作。
- 播放页选择出现时可淡入或上浮。
- 后台面板切换要快速，避免长动画影响效率。
- 保存状态变化应即时反馈。

限制：

- 常规过渡 120-180ms。
- 播放页沉浸转场可到 240ms。
- 后台避免大面积动画。
- 不使用持续旋转、漂浮装饰物或干扰阅读的背景动画。

## 10. 响应式规则

### 10.1 首页

- 桌面端可以使用精选区加作品栅格。
- 移动端优先单列作品流。
- 作品封面保持稳定比例，避免加载后跳动。

### 10.2 播放页

- 桌面端视频区域优先占据主视野。
- 移动端选择区可从底部浮出。
- 横屏移动设备要优先保留视频尺寸。
- 控制层不能遮挡关键选择。

### 10.3 后台

- 桌面端使用多栏工作区。
- 平板端允许折叠侧栏。
- 手机端不作为主要创作环境，但必须能查看、预览和做轻量编辑。

## 11. 可访问性

最低要求：

- 文字和背景对比度足够。
- 按钮有明确可点击区域。
- 图标按钮有可访问名称。
- 表单错误不仅依赖颜色。
- 键盘可操作关键控件。
- 播放页字幕、转写或剧情文本应作为长期目标保留。

## 12. 设计禁区

后续 UI 重构避免：

- 把首页做成纯营销落地页。
- 播放页出现后台编辑控件。
- 后台使用大面积电影暗色导致表单可读性下降。
- 过度使用紫蓝渐变。
- 过度使用单一暖黄、米色、棕橙配色。
- 多层卡片嵌套。
- 用大段说明文字解释功能。
- 使用无法承载长中文的固定宽度按钮。
- 让动态内容改变工具栏、棋盘、图表或编辑面板的基础尺寸。

## 13. 后续落地顺序

1. 先建立页面级布局骨架：首页、播放页、后台分别确定信息密度。
2. 再建立基础组件：按钮、输入、卡片、状态、弹窗、抽屉。
3. 再重构后台编辑器局部 UI，优先保证编辑效率和校验反馈。
4. 再重构播放页叠层、选择、结局和错误状态。
5. 最后优化首页作品发现和视觉呈现。

每次 UI 修改都需要检查桌面和移动端，确保文字不重叠、不溢出，关键操作可见，播放页视频和后台表单各自保持清晰。
