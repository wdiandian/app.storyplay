# 创作后台画布设计（旧方案）

> 当前主方案已升级为 [创作后台画布 V2 方案](./creator-canvas-v2-plan.md)。本文只保留为上一阶段思路参考，不再作为实施依据。

本文档定义 StoryPlay App 创作后台的下一阶段目标形态：从表单工作流转向画布工作流，并为播放页的“回忆 / 线索 / 回响”系统提供可编辑能力。

当前目标不是一次性重写后台，而是先明确产品结构和实施边界，再用低风险步骤把现有后台改造成画布式编辑器。

## 1. 改造目标

创作后台应该让创作者围绕“故事结构”工作，而不是围绕一长串表单工作。

目标体验：

1. 进入项目后，第一眼看到剧情画布。
2. 节点代表剧情片段。
3. 连线代表自动跳转或玩家选择。
4. 点击节点或连线，在右侧检查器编辑。
5. 线索、回响、回忆树显示配置跟随节点、选择和时间线事件编辑。
6. 发布检查问题能直接定位到画布对象。

设计原则：

- 画布是主工作区。
- 表单是属性检查器。
- 游戏性记录系统是核心能力，不是播放器临时面板。
- 先复用现有数据和 API，再逐步扩展模型。

## 2. 当前问题

当前后台已有不少能力：

- 项目设置。
- 节点编辑。
- 选择编辑。
- 时间线事件。
- 变量、条件、动作。
- 旧分支图组件。
- 发布检查类提示。
- 试玩、导入、导出。

但问题是：

- 分支图只是辅助模块，不是主入口。
- 创作者需要先理解大量字段，才能搭出一个故事。
- 节点、选择、时间线、变量和动作混在一个大组件中。
- 播放页的“回忆 / 线索 / 回响”没有正式后台配置入口。
- 时间线 payload 仍然有 JSON 编辑痕迹。

## 3. 目标信息架构

后台目标布局：

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部工具栏：项目、保存、预览、发布检查、导入导出               │
├──────────────┬───────────────────────────────┬───────────────┤
│ 左侧资源栏   │ 中央剧情画布                  │ 右侧检查器    │
│ 项目/素材    │ 节点、连线、问题、解锁状态     │ 当前对象编辑  │
│ 变量/线索    │ 缩放、筛选、快速创建           │ 校验/预览     │
│ 问题列表     │                               │               │
├──────────────┴───────────────────────────────┴───────────────┤
│ 可选底部时间线：当前节点的视频事件、线索、回响触发点           │
└──────────────────────────────────────────────────────────────┘
```

### 3.1 顶部工具栏

职责：

- 切换项目。
- 显示保存状态。
- 预览当前项目。
- 预览当前节点。
- 运行发布检查。
- 导入 / 导出。
- 返回首页。

顶部工具栏只放全局动作，不放节点表单。

### 3.2 左侧资源栏

左侧资源栏负责“找东西”和“看问题”。

首批分组：

- 项目：基础信息入口、首页展示入口。
- 节点：节点列表、搜索、筛选。
- 变量：全局变量和状态定义。
- 线索：线索库。
- 问题：发布检查问题。

左侧不承担大量编辑，只承担导航和快速状态判断。

### 3.3 中央剧情画布

中央画布是主工作区。

画布对象：

- 视频节点。
- 结局节点。
- 自动跳转线。
- 玩家选择线。
- 时间线选择线。
- 问题标记。
- 已解锁 / 可解锁标记。

画布基础能力：

- 点击节点选中。
- 点击连线选中。
- 从节点快速创建后续片段。
- 从节点快速创建玩家选择分支。
- 过滤问题节点、起始节点、结局节点、孤立节点。
- 预览节点。

第一阶段曾计划复用旧分支图组件；该方向已被 V2 画布编辑器取代。

### 3.4 右侧检查器

右侧检查器根据选中对象变化。

未选中对象：

- 项目概览。
- 发布检查摘要。
- 快捷创建节点。
- 最近问题。

选中项目：

- 标题。
- slug。
- 首页展示。
- 宣传视频。
- 宣传文案。
- 起始节点。
- 全局变量入口。

选中节点：

- 节点标题。
- 节点描述。
- 视频地址。
- 正文 / 转写。
- 节点类型。
- 结局设置。
- 回忆树显示设置。
- 节点级线索。
- 当前节点时间线入口。

选中连线：

- 连线类型：自动跳转 / 玩家选择。
- 选择文案。
- 选择提示。
- 目标节点。
- 条件。
- 动作。
- 选择后回响。

选中时间线事件：

- 触发时间。
- 事件类型。
- 显示文本。
- 是否暂停。
- 是否解锁线索。
- 是否产生回响。
- 是否跳转。
- 条件和动作。

## 4. 回忆 / 线索 / 回响系统

播放页现在已经有“剧情记录”面板，但当前内容由现有字段临时推断。后台需要正式支持这些游戏性内容。

### 4.1 回忆

回忆是玩家已经走过的剧情路线。

前台表现：

- 用剧情树展示。
- 已走过节点点亮。
- 已选择分支显示连线。
- 当前节点高亮。
- 未解锁内容可隐藏或显示锁定轮廓。

后台配置：

- 节点在回忆树中的显示名。
- 节点摘要。
- 节点是否进入回忆树。
- 未解锁时是否显示锁定占位。
- 结局节点的回忆展示文案。

建议字段：

```ts
type MemoryConfig = {
  title?: string;
  summary?: string;
  visibleInMemory?: boolean;
  lockedLabel?: string;
};
```

第一阶段不新增字段时：

- 使用节点 `title` 作为回忆标题。
- 使用节点 `description` 作为回忆摘要。
- 使用 `playthrough.history` 和当前节点生成回忆树。

### 4.2 线索

线索是随着剧情推进逐步解锁的信息。

前台表现：

- 在剧情记录的“线索”页签中展示。
- 未解锁线索不显示，或显示为锁定条目。
- 解锁时可以出现短提示。

后台配置：

- 线索标题。
- 线索摘要。
- 线索正文。
- 线索分类。
- 解锁来源：节点、选择、时间线事件、条件。
- 是否在播放页弹出提示。

建议字段：

```ts
type ClueDefinition = {
  id: string;
  title: string;
  summary: string;
  body: string;
  category?: string;
  revealMode: "silent" | "toast" | "overlay";
};

type ClueUnlock = {
  clueId: string;
  sourceType: "node" | "choice" | "timeline_event";
  sourceId: string;
  conditions?: ConditionRule[];
};
```

第一阶段不新增字段时：

- 使用项目 `promoText` 作为开场线索。
- 使用节点 `description` 作为当前片段线索。
- 使用 `show_text` 和 `show_overlay` 作为临时线索来源。

### 4.3 回响

回响是选择或事件造成的后果提示。它不应该直接暴露变量名，而应该把逻辑变化翻译成玩家能理解的剧情反馈。

前台表现：

- 在剧情记录的“回响”页签中展示。
- 选择后可以在转场或短提示中出现。
- 回响可以不解释数值，只表达影响。

后台配置：

- 选择后的回响文案。
- 动作后的回响文案。
- 时间线事件后的回响文案。
- 是否立即弹出。
- 是否只进入记录。

建议字段：

```ts
type EchoDefinition = {
  id: string;
  title: string;
  body: string;
  sourceType: "choice" | "timeline_event" | "action";
  sourceId: string;
  revealMode: "silent" | "toast" | "transition";
};
```

第一阶段不新增字段时：

- 使用选择文案生成选择回响。
- 使用 runtime action payload 生成事件回响。
- 使用变量变化临时生成状态回响。

## 5. 数据模型分阶段策略

不要第一步就做破坏性迁移。

### 阶段 1：临时映射

不改数据模型。

- 回忆来自 `playthrough.history`、`currentNode`、`StoryNode.title`、`StoryNode.description`。
- 线索来自 `promoText`、`node.description`、`show_text`、`show_overlay`。
- 回响来自 `choice.label`、`runtimeActions`、变量预览。

目标：

- 先验证播放页和后台画布的体验。
- 不阻塞当前项目编辑。

### 阶段 2：扩展草稿模型

在 `StoryGame` / `StoryNode` / `StoryChoice` / `TimelineEvent` 上增加可选字段。

建议：

- `StoryGame.clues?: ClueDefinition[]`
- `StoryNode.memory?: MemoryConfig`
- `StoryNode.unlockClueIds?: string[]`
- `StoryChoice.echo?: EchoDefinition`
- `TimelineEvent.unlockClueIds?: string[]`
- `TimelineEvent.echo?: EchoDefinition`

要求：

- 旧数据没有这些字段也必须能正常运行。
- 后台保存时保留未知字段或做安全补齐。

### 阶段 3：发布版本快照

当发布流建立后，回忆、线索、回响必须进入发布快照。

原因：

- 玩家游玩的是发布版本。
- 创作者继续编辑草稿时，不应该改变玩家已开始的记录系统。

## 6. 第一轮后台 UI 改造范围

第一轮只做“画布入口”，不做完整画布引擎。

改造目标：

- 后台打开项目后默认进入剧情画布。
- 旧分支图从辅助区域提升为中央主区域。
- 左侧导航收敛为项目、节点、问题。
- 右侧出现节点检查器雏形。
- 原有完整表单可以暂时保留在检查器或下方高级区。

不做：

- 节点拖拽保存。
- 连线拖拽创建。
- React Flow 引入。
- 数据模型迁移。
- 完整线索库 CRUD。

### 6.1 第一轮布局

```text
┌ 顶部工具栏 ──────────────────────────────────────┐
│ 项目名 / 保存状态 / 预览 / 发布检查 / 返回首页    │
├ 左侧栏 ┬ 中央画布 ┬ 右侧检查器                   │
│ 节点   │ 旧分支图主视图 │ 当前节点快速编辑     │
│ 问题   │ 快速创建后续/分支  │ 视频/标题/出口/试玩   │
│ 变量   │                   │ 高级编辑入口          │
└────────┴──────────┴─────────────────────────────┘
```

### 6.2 第一轮检查器字段

节点检查器先放最关键字段：

- 标题。
- 描述。
- 视频地址。
- 节点类型。
- 是否结局。
- 自动跳转。
- 默认选择列表摘要。
- 时间线事件数量。
- 保存按钮。
- 试玩按钮。

线索和回响先放占位区：

- 回忆显示：使用节点标题和描述。
- 线索：当前阶段从节点描述临时生成。
- 回响：选择文案临时生成。

这样前台和后台的概念先对齐，但不急着做完整字段。

## 7. 后续增强顺序

1. 画布主工作区。
2. 右侧节点检查器。
3. 连线检查器。
4. 底部时间线事件条。
5. 线索库。
6. 回响配置。
7. 回忆树显示配置。
8. 发布检查定位到画布对象。
9. 评估是否引入 React Flow。

## 8. 验收标准

第一轮完成后，应该满足：

- 创作者进入项目后第一眼看到剧情结构。
- 点击画布节点后，右侧能编辑最常用字段。
- 新建后续和新建分支仍可从画布触发。
- 播放页“回忆 / 线索 / 回响”的概念在后台能看到对应占位。
- 原有项目设置、节点编辑、时间线、变量能力不丢失。
- `npm run build` 通过。

## 9. 当前不做

- 不改数据库结构。
- 不引入新画布库。
- 不重写全部后台组件。
- 不做多人协作。
- 不做账号权限。
- 不做完整发布版本系统。
