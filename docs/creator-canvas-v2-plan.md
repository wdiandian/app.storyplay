# 创作后台画布 V2 方案

本文档重新定义 StoryPlay App 创作后台的目标形态。上一版改造把表单搬进画布，但本质仍是“后台表单 + 分支图”。V2 的方向是把创作后台改成真正的互动影游导演台：创作者在画布上组织场景、选择、条件、记录和结局，再由系统编译成播放器可运行的数据。

底层技术实现以 [节点图编辑器底层架构](./node-graph-editor-architecture.md) 为准。本文负责产品方向，节点、端口、连线、编译器和迁移策略以该技术文档为实施依据。

## 1. 核心结论

创作后台不应该是 CMS，也不应该是表单后台。

它应该是：

> 一个用画布编排互动影游的导演台。

创作者的主要工作不是填写字段，而是：

1. 放置场景。
2. 连接剧情走向。
3. 在场景内安排互动时刻。
4. 配置选择带来的后果。
5. 设计玩家逐步解锁的回忆、线索和回响。
6. 随时从当前节点试玩。

## 2. 上一版为什么不对

上一版尝试把编辑能力直接搬到画布节点里，但问题仍然存在：

- 节点里堆了大量表单，画布变成了“横向展开的表单页”。
- 项目、片段、分支、时间线、线索没有成为独立画布对象。
- 创作者仍然需要理解现有 `node -> choices -> timelineEvents` 的技术结构。
- 画布没有提供“互动影游创作”的专属对象，例如选择点、条件点、记录点。
- 回忆 / 线索 / 回响仍然是从现有字段推断出来的展示块，不是可设计的系统。

所以 V2 不再以“把字段放到哪里”为中心，而是先重建创作模型。

## 3. 产品模型

V2 后台的核心对象不是“表单字段”，而是画布对象。

### 3.1 画布对象

首批对象：

| 对象 | 作用 | 示例 |
| --- | --- | --- |
| 入口节点 | 作品开场、开始按钮、首个落点 | 开场说明、进入故事 |
| 场景节点 | 一段视频或一场戏 | 电梯间、监控室 |
| 选择节点 | 玩家做决定的位置 | 追上去 / 留在原地 |
| 条件节点 | 判断变量、线索、状态 | 已拿到门禁卡 |
| 记录节点 | 回忆、线索、回响的解锁内容 | 电梯里的血迹、陌生来电 |
| 结局节点 | 结局和结算 | 逃出生天、困在循环 |
| 素材节点 | 视频、图片、音频、字幕资源 | elevator.mp4 |

这些对象在画布上可独立选择、连接和编辑。

### 3.2 连接关系

连线不是简单的跳转线，而是剧情关系。

首批连线：

- `进入`：入口到首场景。
- `播放后进入`：场景自然结束后进入下一个对象。
- `选择后进入`：选择节点指向目标场景或结局。
- `满足条件进入`：条件成立时进入目标。
- `解锁记录`：场景、选择或条件触发记录节点。
- `使用素材`：场景绑定素材节点。

连线可以被选中，右侧属性面板显示它的类型、条件、动作和玩家侧文案。

## 4. 界面结构

V2 界面保持三层，但不是旧三栏后台。

### 4.1 顶部导演栏

只放全局动作：

- 项目切换。
- 保存状态。
- 试玩当前节点。
- 试玩完整作品。
- 发布检查。
- 素材库。
- 导入 / 导出。

顶部不放节点字段。

### 4.2 中央无限画布

画布是主工作区。

能力：

- 拖拽放置节点。
- 从节点拖出连线。
- 缩放、平移、框选。
- 按问题、角色、章节、结局筛选。
- 节点上显示摘要和问题状态。
- 节点不展开成长表单。

节点显示原则：

- 只显示结构信息。
- 只显示创作者做判断需要的状态。
- 详细编辑进入属性面板。

### 4.3 右侧属性面板

选中什么，编辑什么。

未选中：

- 项目概览。
- 发布检查摘要。
- 最近编辑。
- 快捷创建。

选中场景节点：

- 场景标题。
- 视频素材。
- 场景摘要。
- 片内互动时间轴入口。
- 自然结束后的流向。
- 试玩当前场景。

选中选择节点：

- 选择题干。
- 选项文案。
- 目标节点。
- 条件。
- 变量影响。
- 选择后回响。

选中记录节点：

- 类型：回忆 / 线索 / 回响。
- 玩家侧标题。
- 展示正文。
- 解锁来源。
- 未解锁展示方式。
- 是否进入播放页剧情记录。

选中条件节点：

- 条件变量。
- 判断规则。
- 成立 / 不成立的出口。

选中连线：

- 连线类型。
- 玩家侧文案。
- 条件。
- 动作。
- 是否解锁记录。

## 5. 回忆 / 线索 / 回响

这部分应该从播放器临时面板升级为创作后台的一等对象。

### 5.1 回忆

回忆不是普通字段，而是玩家走过路径的可视化结果。

后台配置：

- 哪些场景进入回忆树。
- 节点在回忆树中的标题和摘要。
- 未解锁节点是否显示轮廓。
- 结局节点的回忆说明。

### 5.2 线索

线索是被剧情逐步揭开的信息。

后台配置：

- 线索标题。
- 线索正文。
- 解锁来源：场景、选择、时间点、条件。
- 是否有锁定态。
- 是否关联变量或后续条件。

### 5.3 回响

回响是玩家选择造成的后果感。

后台配置：

- 回响标题。
- 回响文案。
- 触发来源：选择、条件、结局。
- 影响变量。
- 是否在播放页剧情记录中展示。

## 6. 数据模型方向

当前运行数据接近：

```text
Game
  └─ StoryNode
       ├─ choices
       └─ timelineEvents
```

这个结构可以运行，但不适合作为创作模型。

V2 应该拆成两层：

### 6.1 创作模型

用于后台编辑。

```ts
type CanvasObject =
  | EntryObject
  | SceneObject
  | DecisionObject
  | ConditionObject
  | RecordObject
  | EndingObject
  | AssetObject;

type CanvasEdge = {
  id: string;
  fromObjectId: string;
  toObjectId: string;
  type: "enter" | "auto" | "choice" | "condition" | "unlock_record" | "use_asset";
  label?: string;
  conditions?: ConditionRule[];
  actions?: VariableAction[];
};
```

### 6.2 运行模型

用于播放器。

现阶段播放器仍可继续使用现有 `StoryGame / StoryNode` 结构。

后续由创作模型编译成运行模型：

```text
CanvasProject -> StoryGame
```

这样后台可以越来越像专业创作工具，播放器仍保持稳定。

## 7. 分阶段路线

### 阶段 1：画布原型，不改运行数据

目标：验证新的创作体验。

范围：

- 新建独立 `CanvasEditor` 组件。
- 用现有 `StoryGame` 生成画布对象。
- 场景节点、选择节点、结局节点可视化。
- 点击对象后在右侧属性面板编辑。
- 保存仍调用现有 API。
- 不再在画布节点里展开大表单。

验收：

- 进入后台第一眼看到画布。
- 创作者能清楚看到“场景 -> 选择 -> 场景 / 结局”。
- 选中对象后只编辑该对象的属性。

### 阶段 2：记录节点进入画布

目标：把回忆 / 线索 / 回响做成可设计对象。

范围：

- 新增记录节点展示。
- 场景、选择、时间点可以连接记录节点。
- 播放页剧情记录从后台配置读取，而不是临时推断。

验收：

- 创作者能明确设置某条线索何时解锁。
- 播放页能展示后台配置的线索和回响。

### 阶段 3：时间轴子编辑器

目标：让场景内互动不再靠 JSON 或长表单。

范围：

- 选中场景后打开场景时间轴。
- 在时间轴上添加提示、暂停选择、解锁记录、跳转。
- 时间轴事件以视觉标记呈现。

验收：

- 创作者能在某个视频时间点插入互动。
- 不需要手写 payload。

### 阶段 4：创作模型与运行模型分离

目标：后台进入真正的节点编辑器架构。

范围：

- 存储 `CanvasProject`。
- 编译输出 `StoryGame`。
- 发布检查基于 CanvasProject。
- 播放器只消费编译后的运行数据。

验收：

- 创作后台可以新增画布对象类型，不直接影响播放器。
- 运行时数据保持稳定。

## 8. 当前不要做什么

为了避免继续小修小补，下一步不要做：

- 不要继续把字段塞进现有 `admin-story-editor.tsx`。
- 不要把节点变成超大表单卡片。
- 不要在旧 `BranchGraph` 上继续堆复杂编辑。
- 不要急着新增数据库字段。
- 不要先做美术化皮肤。

应该先做：

- 独立 CanvasEditor 原型。
- 明确对象类型和属性面板。
- 用现有数据生成画布视图。
- 让保存链路继续复用现有接口。

## 9. 下一步执行建议

下一步不是继续改当前后台，而是新建 V2 原型：

1. 新建 `src/components/admin-canvas-editor.tsx`。
2. 从 `StoryGame` 派生 `CanvasObject[]` 和 `CanvasEdge[]`。
3. 实现中央画布 + 右侧属性面板。
4. 只接入场景节点、选择节点、结局节点。
5. 确认体验方向后，再替换 `/admin` 主入口。

这一步的目标是产品体验验证，不是一次性完成所有后台能力。
