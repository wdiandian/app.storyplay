# 2026-06 前台大改版更新

## 本轮目标

这一轮不再继续做后台或规则系统，而是集中处理玩家端体验，把现有前台从“视频页面 + 选项卡片”推进到更接近互动影游的舞台式呈现。

本轮重点：

- 强化作品入口页
- 强化播放中的舞台感
- 强化关键交互时刻的演出感
- 保持现有后端接口和运行时协议不变

## 已完成内容

### 1. 宣传页改成正式作品入口

玩家进入首页后，不再直接落到普通播放器视图，而是先进入一个更像“发售作品入口”的宣传页：

- 左侧主视觉使用宣传视频 / 当前节点视频作为大画面
- 作品标题、标语直接压在画面上
- 右侧使用独立入口卡片承接作品导语和首幕信息
- 提供明确的“开始游玩”入口

这一步的目的不是只做一页封面，而是把“进入作品”与“进入某一幕场景”分成两个层级。

### 2. 播放器改成舞台优先结构

播放器主体已经收敛为“视频舞台优先”的结构：

- 主体几乎就是舞台画面，不再保留明显的页面式头部
- 信息区改成右侧抽屉面板，默认收起
- 当前场景信息改成 HUD 浮层，而不是独立内容区
- 选项层压在视频舞台底部，默认与场景融为一体

这样处理之后，用户感知会更像在“玩一段剧情”，而不是在“操作一个后台感很强的页面播放器”。

### 3. 节点开始演出拆成独立层

这次把“进入新节点”的视觉反馈拆成了两段：

- `Chapter Transition`
  用于强调章节/幕切换
- `Scene Arrival`
  用于强调具体场景登场

前者是更强的章节切换感，后者是更贴近镜头开场的场景入场提示。

这样比单一转场提示更适合互动影游，因为玩家会同时感知“叙事章节变化”和“场景镜头开始”。

### 4. 决策时刻拆成独立演出层

选项出现不再只是把按钮渲染出来，而是新增了单独的“决策时刻”反馈：

- 画面整体压暗
- 屏幕上方短暂出现 `Decision Point`
- 选项卡片带有独立入场动画
- 不同选项类型保留不同色彩倾向和轻微发光

这一步的目标是让“做选择”变成一个被强调的时刻，而不是普通表单操作。

### 5. 结局达成演出继续加强

结局区域已经从普通结尾文案，推进成更完整的“结局达成”层：

- 结局标题与结局类型独立展示
- 使用结局倾向色做背景氛围
- 补充路径回声与累计分支等摘要信息
- 为结局主体和摘要增加分段入场动画

目前这已经具备基础的“达成结局”仪式感，后续可以继续往结局画廊方向扩展。

### 6. 保持现有运行时逻辑稳定

虽然本轮是前台重做，但没有改动接口协议，也没有推翻之前的规则系统联动逻辑：

- 仍然沿用现有 `playthrough` 接口
- 仍然支持时间线事件触发选项、文本、叠层、跳转、动作
- 仍然保留“播放完停留最后一帧”的处理
- 仍然保留时间线事件只同步 `playthrough` 状态、不整页重置 UI 的策略

这意味着前台表现层和运行时逻辑现在是分离的，后续继续加表现不会牵动规则协议。

## 当前结论

到这一步，前台已经从“可播放的互动视频页”进入到“有明显作品感和关键时刻演出感的互动影游前台”。

但它还不是最终形态，原因有三点：

1. 目前的演出仍然是基于页面层和 CSS 动画，尚未形成更完整的镜头语言系统。
2. 选项虽然已经更像游戏内决策，但还没有和音效、节奏点、镜头停顿形成联动。
3. 结局页已经比之前完整很多，但还没有做成可沉淀、可回顾、可收藏的“结局资产页”。

## 下一阶段建议

建议下一阶段按下面顺序推进：

1. 做前台“关键时刻演出系统化”
   把节点开始、分支选择、结局达成、自动跳转，分别定义成统一可配置的演出模板，而不是继续只写死在组件里。

2. 做移动端专项沉浸优化
   重点处理竖屏下的选项布局、HUD 密度、按钮触达和视频控件干扰。

3. 做音频与反馈层
   包括分支出现提示音、选项确认反馈、结局达成音效、轻微环境氛围层。

4. 做结局沉淀页
   把“结局摘要”升级成可回看、可展示、可分享的结局结果页。

5. 做前台皮肤/主题抽象
   如果后续目标真的是平台化，而不是只服务一个作品，就需要把前台的视觉层做成可以切换主题的结构。
